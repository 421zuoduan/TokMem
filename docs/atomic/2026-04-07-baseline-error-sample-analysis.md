## 2. 一句话结论

这条 baseline 的主要问题不是“首个任务标记大面积选错”，而是“首个任务标记经常已经对了，但后面又继续生成别的任务标记，导致最终 `task_match=False`”。

按新表的口径看，最关键的几件事是：

- `32460` 个样本是完全正常的：首个任务标记正确，而且只生成了一个任务标记
- `1273` 个样本是：首个任务标记已经对了，但后面又出现了其他任务标记
- `668` 个样本是：首个任务标记先错到同族任务，其中 `597` 个后面一直没回正，`71` 个后面回到了正确任务
- `334` 个样本是：首个任务标记先错到相似任务，其中 `317` 个后面一直没回正，`17` 个后面回到了正确任务
- `162` 个样本是：首个任务标记一开始就错到明显不相近的任务，其中只有 `5` 个后面回到了正确任务
- `23` 个样本是：完全没有生成任务标记

也就是说：

- 最大头仍然是“首个任务标记对了，但后面又多生成了别的任务标记”
- 真正的“首个任务标记一开始就选错”一共有 `1164` 个，其中大多数还是先错到同族或相似任务，而不是直接跑得很远

## 3. 先看错误机制

### 3.1 多任务标记溢出是主导错误

- 多于一个 `predicted_task` 的样本总共有 `1597` 个
- 其中只有 `80` 个最终 `task_match=True`
- 多任务标记样本的任务准确率只有 `5.01%`
- 单任务标记样本的任务准确率是 `97.24%`

这说明当前 baseline 最脆弱的点，是“任务标记序列停不住”。

更细一点看：

- `1273` 个错误样本里，第一跳其实已经对了，但后面又吐出了额外任务标记
- 这部分占全部错误样本 `51.75%`
- 这部分样本里空响应比例高达 `68.19%`
- 这部分样本平均 `ROUGE-L` 只有 `0.0740`

这类错误一旦发生，通常不只是路由判错，回答本身也会被严重破坏。

### 3.2 256 长度跑满不是个别现象

在错误样本里，`predicted_tasks` 长度分布的头部是：

- 长度 `2`: `756`
- 长度 `3`: `215`
- 长度 `256`: `137`
- 长度 `4`: `91`
- 长度 `255`: `28`
- 长度 `253`: `16`

这说明除了常见的“多吐一个同簇任务”，还存在一批明显的任务标记失控解码。

### 3.3 空响应和新表里的几类模式高度相关

- 全部错误样本里，空响应有 `1093` 个，占 `44.43%`

这里的“空响应”，指的是代码里先把模型输出解码，再抽取第一段回答文本并做 `strip()` 之后，结果是空字符串。它不等于“模型只输出了 memory token”。更常见的情况是：任务标记彼此贴得太近，中间没有正常回答文本；或者任务标记后面立刻就是 `EOS` / 其他会被清掉的特殊标记；或者第一段回答在切分后已经变成空串。

按新表里的关键几类拆开：

| 模式 | 空响应占比 | 平均 ROUGE-L |
| --- | ---: | ---: |
| 首个任务标记正确，但后续出现了其他任务标记 | `68.19%` | `0.0740` |
| 首个任务标记错误，首个错到同族任务，且后续没有正确任务标记 | `8.54%` | `0.3969` |
| 首个任务标记错误，首个错到相似任务，且后续没有正确任务标记 | `14.51%` | `0.3126` |
| 首个任务标记错误，首个既不属于同族也不属于相似任务，且后续没有正确任务标记 | `25.48%` | `0.1799` |
| 完全没有生成任务标记 | `0.00%` | `0.5762` |

可以看到：

- 真正最伤回答质量的，仍然是“首个任务标记已经对了，但后面又多生成了别的任务标记”
- 纯粹的“首个任务标记先错到同族任务、而且后面没回正”，回答质量反而没有想象中那么差
- 一旦首个任务标记先错到更远的任务，平均 ROUGE-L 会进一步下降

## 7. 对这条 baseline 的判断

### 7.1 当前最核心的问题

当前 baseline 的首要问题不是“首个任务标记完全不会选”，而是：

- 首个任务标记经常已经选对了，但后面又继续生成了别的任务标记

证据很明确：

- `1273` 个错误样本里，首个任务标记本来就是对的，但后面又出现了别的任务标记
- 多任务标记样本的任务准确率只有 `5.01%`
- 这类错误伴随极高的空响应和极低的 ROUGE-L

### 7.2 第二层问题

在“首个任务标记一开始就选错”的样本里，大多数也不是“直接跑到很远的任务”，而是：

- `668` 个样本先错到同族任务
- `334` 个样本先错到相似任务
- 只有 `162` 个样本一开始就错到既不属于同族也不属于相似任务的更远任务

更重要的是，这些首个任务标记错误大多不会自己纠正回来：

- 同族任务这类里，只有 `71 / 668` 个样本后面回到了正确任务
- 相似任务这类里，只有 `17 / 334` 个样本后面回到了正确任务
- 更远任务这类里，只有 `5 / 162` 个样本后面回到了正确任务

典型就是：

- `jeopardy`
- `scan`
- `anli`
- `winomt`
- `record`

这些簇的共同点是：

- 指令近乎一样
- 差异更多来自数据切片而不是任务规则

### 7.3 第三层问题

如果只看“首个任务标记先错到相似任务或更远任务，而且后面没有回正”的这两类，一共有：

- `317 + 157 = 474`

其中 QA 占了最大头。也就是说，除了同族任务切分混淆之外，QA 家族本身的可分性也确实不足。

## 8. 最值得优先处理的方向

如果目标是继续提升这条 baseline，优先级应该是：

1. 先解决解码期多任务标记溢出
   - 这是最大错误来源，直接关系到一半以上的错误样本
2. 再针对“首个任务标记容易先错到同族任务”的那几簇做区分增强
   - 重点是 `jeopardy / scan / anli / winomt / record`
3. 最后再处理“首个任务标记先错到相似任务或更远任务”的 QA 长尾
   - 这里更接近真正的路由能力边界

换句话说，这次 baseline 的错误结构不是“一个问题”，而是三层叠加：

1. `首个任务标记已经对了，但后续仍会继续生成其他任务标记`
2. `首个任务标记容易先错到同族任务，而且后面通常回不来`
3. `QA 里有一批样本，首个任务标记会先错到相似任务或更远任务`

其中第一层最该先动。
