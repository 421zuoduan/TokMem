# TokMem 梳理

Atomic 实验里, 一个 memory token 对应一个 procedure/task; Compositional 实验里, 一个 memory token 对应一个 tool, 输出 tool + 参数 (arguments), 回答仅包括 tool 和参数, 所以论文里表三给的是 F1 Score


## Atomic 实验

只需预测一个 memory token, 一个 memory token 路由到一个 procedure/task

- 训练数据形式: `<m> + response`, 仅使用 CE loss 训练
- 数据量: 700 task, 每个 task 有 500 train + 10 val + 50 test

### 组合数据集

选取高频 tool, 利用这些 tool 合成 Compositional 数据

- 训练数据:
    - 真实数据: 单/多次调用样本直接当作训练数据, 共 50tool × 50 条数据 
    - 合成数据: 
        1. 随机选 2/3 个 tool, 每个 tool 取一个 query-call pair (这里叫 call 是因为可能是单次调用也可能是多次调用)
        2. 使用连接词拼接多个 query, 按顺序拼接 tools 和 calls. 要求 function call 次数最大为 4
        3. 2tool 和 3tool 各 1250 条
- 测试数据合成同上, 2tool 和 3tool 各 250 条, 无单 tool 数据

问题:

## 训练过程

纯 CE loss, 进入下一个 memory token 前没有一个标志上一个 memory token 控制生成结束的 token



## 思路

routing 的问题: 没有一个标志上一个 memory token 控制范围内生成结束的 token, 可以加一个控制生成结束的 token, 将这个路由显式化控制, 这样也能加一个 routing loss

任务形式的问题: 如果换成真正调用工具去完成任务的任务, 就不用改输出形式, 也就不需要 adaptation

1. 加入三个 mode token, 用以控制输出模式 (需要有参方法): 既有 adaptation 学习的是 `\<memory1><arguments1><memory2><arguments2>` 这种输出格式, 仅 tokmem 也能一定程度上学习这种输出格式, 但是词表里没有专门用来学习这种输出格式的 token --> 额外添加三个可学习 token, 数据做成先生成这三个 token, 再继续生成 `\<memory1><arguments1><memory2><arguments2>`. 看实验效果可以加重这部分的 loss
2. 添加 `RET` token: 添加 memory token 控制范围的显式约束, `RET` 后可以是 memory token 或自然语言文本 (有自然语言文本是考虑比较复杂的场景下, 不让约束过强), 训练时加一个 `RET` 预测的 loss, 显式约束输出方式 (某种程度上这也是解耦)
3. 基于 2, 可以把 atomic 的任务混合进来: 如果模型输出了前面的 mode token, 代表他是要执行 tool 调用的任务; 如果没有生成 mode token, 可以直接生成 atomic 里的 memory token
4. decoupled embedding: decoupled embedding 是把词表的 input_embedding 和 output embedding 解耦了
5. 现在原始数据里还有一部分没有用到, 比如 decription, 能不能把 decription 过一个 encoder 加到 steering embedding 上
